SECTION 5 — V2 CHANGES: HOW TO IMPLEMENT IN CODE

5.1  Partner Login (ID + Password) — Backend
In routes/auth.js:
POST /api/auth/partner/login
  → Receive username + password
  → Find partner by username in DB
  → bcrypt.compare(password, partner.password_hash)
  → If match: issue JWT with { partnerId, role: 'partner' }
  → If no match: return 401 Unauthorized

In middleware/auth.js — protect partner routes:
  → Verify JWT → check role === 'partner'
  → Add req.partnerId = token.partnerId
  → In GET /api/partner/jobs: WHERE partner_id = req.partnerId
  → Partner sees ONLY their own jobs — cannot see others

5.2  Cloth Check Before Pre-Payment — Logic Flow
Step
Old V1 Order
New V2 Order (Your Correction)
1
Partner taps Reached
Partner taps Reached
2
Partner counts clothes
Partner checks cloth condition ← NEW POSITION
3
Payment request sent
Customer accepts/auto-accepts condition
4
Customer pays
Partner counts actual clothes
5
Partner taps Picked
Payment request sent to customer
6
—
Customer pays (Razorpay or cash)
7
—
Partner taps Picked (unlocked)

Why this order is correct: Cloth condition must be documented and customer must accept BEFORE the cloth count and payment. If a dispute is raised after payment, it becomes harder to resolve. Cloth check before payment = clean record before any money changes hands.
